Product Definition

A Workcell is a project-scoped runtime package for one AI agent instance. It is designed for mobile development where Jared may run several ChatGPT projects, Claude projects, and Perplexity spaces at once. Each instance gets only the tools, instructions, and inbox it needs for its lane.

Tool = one capability Tool Belt = coordinated tools Workcell = belt + instructions + inbox Project Mesh = multiple workcells

Why Workcells Exist

1
Models need tools.
The MCP belt gives the agent hands.
2
Tools need rules.
Boot instructions tell the agent when and how to use those hands.
3
Agents need lanes.
Identity and responsibilities keep multiple sessions from colliding.
4
Lanes need mail.
Inbox/outbox/log files let ChatGPT, Claude, Alice, and Jared coordinate asynchronously.

Tool → Instruction Fragments

Each selected tool contributes operating rules into the generated boot file.

github.commit_file:
- Read before overwriting.
- Commit small reviewable changes.
- Report path and commit SHA.

cloudflare.deploy_worker:
- Deploy only when explicitly asked.
- Verify /health after deploy.
- Check bindings before debugging code.

bridge.send_message:
- Use the standard message header.
- Include from, to, project, priority, requires.
- Route blockers to the correct inbox.

Generator Inputs

  • Project name
  • Assistant target
  • Role / lane
  • Selected tool belt
  • Instruction style
  • Mailbox location
  • Expiration / safety level

Safety Rules

  • Small scoped tool lists.
  • No destructive defaults.
  • Explicit approval for deletes, drops, secrets, and billing.
  • Every workcell has expiration and revocation.
  • Logs and message trail are mandatory.

UX Principle

  • One MCP link.
  • One boot link.
  • One inbox path.
  • One role card.
  • Everything copy/pasteable from iPhone.
