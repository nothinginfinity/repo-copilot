# Code-Icles — Concept Spec v0.1

> Captured: 2026-05-09 | Status: Concept | Owner: Jared
> A Code-Icle is a piece of content that's made from code.

---

## The Core Idea

Every piece of content — article, video, image, application, data visualization — is **generated at view time from code**.

Instead of:
- A writer typing an article → you read static text
- A videographer recording footage → you watch a static file
- A designer exporting an image → you see a static PNG

A Code-Icle is:
- A prompt + LLM + code → generates the article **when you open it**
- A prompt + generative pipeline → renders the video **when you play it**
- A dataset + Python → draws the visualization **when you view it**
- A spec + HTML/CSS/JS → runs the interactive experience **when you click it**

The artifact IS the code. The experience is what the code produces.

---

## Etymology

**Code + Article = Code-Icle**

But the format extends beyond articles:
- Code-Icle (article format) — prose/essay generated from code
- Code-Icle (visual format) — image/infographic generated from code
- Code-Icle (video format) — video/animation generated from code
- Code-Icle (app format) — interactive application (same as Notion App Store tier)

All formats share one property: **the source of truth is code, not a rendered file.**

---

## Why This Is Different From Existing Formats

| Format today | Problem | Code-Icle solution |
|---|---|---|
| Blog article | Static. Stale the moment it's published. | Regenerates on open. Can incorporate live data, user context, current date. |
| YouTube video | Recorded once. Can't adapt to viewer. | Generated from prompts + assets. Each render can be personalized. |
| Instagram image | A JPEG. Fixed forever. | A generative pipeline. The image evolves with the prompt. |
| App store app | Binary. Requires install. | HTML/JS at a URL. Runs anywhere. Updates on push. |
| Newsletter | Text email. | A Code-Icle that renders as email HTML, web page, or audio on demand. |

## Code-Icle Anatomy

Every Code-Icle has:

```
codeicle/
├── manifest.json       # title, type, author, version, tags
├── prompt.md           # the human-language intent (what it should convey)
├── generator.py/.js    # the code that produces the artifact
├── assets/             # any static inputs (data, images, audio)
└── output/             # cached last render (for fast load / offline)
```

### manifest.json
```json
{
  "title": "Why the future of content is code",
  "type": "article",         // article | visual | video | app | data
  "author": "jared",
  "version": "1.0.0",
  "generator": "generator.py",
  "tags": ["meta", "code-icles", "future-of-content"],
  "render_mode": "on_open"   // on_open | cached | scheduled
}
```

## Content Types

### Article Code-Icle
- `generator.py` calls an LLM with the prompt + any live context (date, user, trending data)
- Output: HTML rendered as an article in the browser
- Reader experience: clicks the title in Notion, article renders in ~2 seconds
- Feels like reading. Is actually running code.

### Visual Code-Icle
- `generator.py` produces a chart, infographic, or generative art piece
- Output: SVG, Canvas, or WebGL rendered in the browser
- The "image" is never saved — it's drawn fresh each time

### Video Code-Icle
- `generator.py` assembles frames, narration, and animation from prompts + assets
- Output: rendered MP4 or streaming WebGL scene
- Not recorded footage — a directed pipeline that produces footage

### Data Code-Icle
- `generator.py` pulls live data and renders a visualization
- Output: interactive chart/graph/table
- The content updates every time you view it

### App Code-Icle
- Same as Notion App Store apps — the overlap is intentional
- An app IS a Code-Icle with `type: app`

## Distribution (via Notion + GitHub)

1. Creator writes a prompt describing what they want to convey
2. LLM (repo-copilot) generates the `generator.py` + `manifest.json`
3. Code is pushed to GitHub — gets a GitHub Pages URL
4. `sync-live-sites.yml` auto-populates the URL into Notion
5. Viewer sees a Notion row: title + "Open" button
6. Clicks Open → generator runs → content renders in browser

**The creator never writes a single line of code.**
**The viewer never installs anything.**
**The content is always current.**

## Phase 1 Build — Proof of Concept

The smallest thing that proves the concept:

1. **One Article Code-Icle** — `generator.py` calls an LLM, outputs styled HTML. Hosted on GitHub Pages. Listed in Notion. Click to read.
2. **One Visual Code-Icle** — `generator.py` produces a D3 or Canvas visualization. Renders on open.
3. **`codeicle-runner` repo** — the standard wrapper that loads any Code-Icle manifest and runs its generator. This is the runtime.

## Open Questions

- Should the generator run server-side (GitHub Actions) or client-side (browser JS/WASM)?
  - Client-side: no server cost, works offline, faster iteration
  - Server-side: can use Python/full LLM APIs, richer output
  - **Likely answer: client-side for visual/app types, server-side for article/video types**
- How does a viewer "pay to open" a Code-Icle? (access token in URL? Notion permission?)
- What's the canonical viewer URL pattern? `codeicles.io/{author}/{slug}`?
- Should Code-Icles be forkable? (viewer clicks "remix" → gets a copy of the generator they can modify)

## Connection to Larger Vision

> “An article of the future that's written from my words talking to a large language model
> is eventually going to turn into code and it's gonna be viewed from code.” — Jared, 2026-05-09

This is the thesis:
- **Today:** human writes → human reads
- **Transition:** human prompts LLM → LLM writes → human reads
- **Future (Code-Icles):** human prompts LLM → LLM generates code → code generates content → human views rendered artifact

The content format of the AI era is not text. It's not video. **It's code that renders into whatever the viewer needs.**
